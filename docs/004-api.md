## 공통 규칙
- Base URL: `/api`
- 요청 및 응답 형식: `application/json`
- 날짜 형식: `yyyy-MM-dd`
- 날짜·시간 형식: UTC ISO 8601, 초 단위 (`yyyy-MM-dd'T'HH:mm:ss'Z'`)
- `visitDate`는 업무 날짜이므로 시각을 포함하지 않음
- `createdAt`, `updatedAt` 등 시스템 시각에는 정렬·감사·재시도를 위해 시·분·초를 포함하고 밀리초는 사용하지 않음
- ID 타입: `BIGINT`
- 점수 범위: `0` 이상 `10` 이하
- 점수 타입: 정수
- 성공 응답은 공통 Wrapper 없이 리소스를 직접 반환
- 오류 응답은 공통 형식 사용
- Spring Boot Actuator 운영 Endpoint는 `/api` Base URL과 공통 오류 형식을 사용하지 않음

## 공통 HTTP Status
| HTTP Status | 사용 기준|
|----|----|
| `200 OK` | 조회 및 수정 성공 |
| `201 Created` | 리소스 생성 성공 |
| `204 No Content` | 삭제 성공 |
| `400 Bad Request` | 요청 형식 오류 또는 Validation 실패 |
| `401 Unauthorized` | LINE Webhook 요청 서명 검증 실패 |
| `404 Not Found` | 요청한 리소스가 존재하지 않음 |
| `405 Method Not Allowed` | 지원하지 않는 HTTP Method 요청 |
| `409 Conflict` | 리소스 상태 또는 데이터 충돌 |
| `500 Internal Server Error` | 서버 내부 오류 |
| `502 Bad Gateway` | 외부 API 호출 실패 |
| `503 Service Unavailable` | 외부 서비스 또는 저장소 일시 사용 불가 |

## 오류 응답

```json
{
  "timestamp": "2026-07-12T14:30:00Z",
  "status":400,
  "code": "VALIDATION_FAILED",
  "message": "요청 값이 올바르지 않습니다.",
  "path": "/api/visits",
  "errors":[
    {
      "field": "atmosphereScore",
      "reason": "0이상 10이하의 값이어야 합니다."
    }
  ]
}
```
- `errors`는 필드 단위 오류가 있는 경우에만 포함
- `code`는 Backend에서 정의한 오류 코드 사용

## 공통 오류 코드

| HTTP Status | code | 사용 기준 |
|----|----|----|
| `400` | `MALFORMED_REQUEST` | JSON 문법 오류, 타입 불일치 등 요청 본문을 읽을 수 없음 |
| `400` | `VALIDATION_FAILED` | 필수 값 누락, 길이·범위·날짜 형식 등 필드 검증 실패 |
| `400` | `INVALID_DATE_RANGE` | 조회 시작일이 종료일보다 늦음 |
| `400` | `INVALID_REPORT_TARGETS` | Report Type에 맞지 않는 `areaIds` 개수, 누락, 중복 또는 불필요한 전달 |
| `400` | `UNSUPPORTED_REPORT_TYPE` | 지원하지 않는 Report Type 요청 |
| `401` | `INVALID_LINE_SIGNATURE` | LINE Webhook 요청 서명 검증 실패 |
| `404` | `AREA_NOT_FOUND` | Area가 존재하지 않거나 Soft Delete됨 |
| `404` | `VISIT_NOT_FOUND` | Visit이 존재하지 않음 |
| `404` | `REPORT_NOT_FOUND` | Report가 존재하지 않음 |
| `405` | `METHOD_NOT_ALLOWED` | 해당 경로에서 지원하지 않는 HTTP Method 요청 |
| `409` | `AREA_ALREADY_EXISTS` | `(prefecture, city, name)`이 동일한 Area가 이미 존재 |
| `500` | `INTERNAL_SERVER_ERROR` | 별도 처리되지 않은 서버 내부 오류 |
| `502` | `OPENAI_API_ERROR` | OpenAI API 호출 또는 응답 처리 실패 |
| `503` | `STORAGE_ERROR` | Cloud Storage 저장·조회·삭제 실패 |
| `503` | `LINE_EVENT_DISPATCH_ERROR` | LINE 이벤트를 비동기 처리기로 전달하지 못함 |

- Soft Delete된 Area는 외부에서 존재하지 않는 리소스와 동일하게 `AREA_NOT_FOUND`로 처리한다.
- 오류 `message`는 사용자에게 표시할 수 있는 문장으로 작성하고, 내부 예외 메시지나 민감한 정보를 노출하지 않는다.
- 필드 검증 오류가 여러 개이면 `errors` 배열에 가능한 오류를 함께 반환한다.


## Area API

### 필드 검증 규칙

| 필드 | 필수 | 규칙 |
|----|----|----|
| `name` | 필수 | 문자열, 최대 25자 |
| `prefecture` | 필수 | 문자열, 최대 20자 |
| `city` | 필수 | 문자열, 최대 20자 |
| `station` | 선택 | 문자열, 최대 50자 |

- `prefecture`, `city`, `name`의 조합은 중복될 수 없다.
- 등록 또는 수정 결과 동일한 조합의 다른 Area가 존재하면 `409 Conflict`와 `AREA_ALREADY_EXISTS`를 반환한다.
- 중복 여부를 판단할 때 `station`은 사용하지 않는다. 하나의 지역에 여러 역이 연결될 수 있기 때문이다.

### 지역 삭제
```http
DELETE /api/areas/{areaId}
```

```http
204 No Content
```

- V1에서는 Area를 물리 삭제하지 않고 Soft Delete한다.
- Soft Delete에는 `deleted_at TIMESTAMP NULL` 컬럼을 사용한다.
- 삭제된 Area는 지역 목록 및 상세 조회에서 노출하지 않는다.
- 삭제된 Area에는 새로운 Visit을 등록하거나 Report 생성 대상으로 지정할 수 없다.
- 기존 Visit은 원본 데이터 보존을 위해 유지한다.
- 삭제되지 않은 Area를 삭제하면 `204 No Content`를 반환한다.
- 이미 삭제된 Area를 다시 삭제하거나 조회 또는 수정하면 `404 Not Found`를 반환한다.
- 복구 API와 보존 기간 이후 Hard Delete 정책은 V2에서 검토한다.

### 필드 검증 규칙

| 필드 | 필수 | 규칙 |

| `atmosphereScore` | 필수 | 정수, 0 이상 10 이하 |

## Visit Draft API

자연어 입력을 Visit 등록 요청 형태의 초안으로 변환한다. 이 API는 Visit을 저장하지 않으며, 사용자가 결과를 확인한 후 `POST /api/visits`로 등록한다.

- `text`는 필수이며 빈 문자열은 허용하지 않는다.
- OpenAI API 호출 또는 응답 처리에 실패하면 `502 Bad Gateway`와 `OPENAI_API_ERROR`를 반환한다.

### 자연어 방문 평가 파싱
```http
POST /api/visit-drafts
```

Request
```json
{
  "text": "7월 12일 센터미나미에 갔는데 분위기 9점, 접근성 7점이었어."
}
```

Response
```http
200 OK
```
```json
{
  "area": {
    "id": 1,
    "name": "센터미나미"
  },
  "visitDate": "2026-07-12",
  "atmosphereScore": 9,
  "infraScore": null,
  "cleanScore": null,
  "sizeScore": null,
  "accessScore": 7,
  "memo": null,
  "warnings": [
    "infraScore, cleanScore, sizeScore가 입력되지 않았습니다."
  ]
}
```

- 파싱 결과에는 불확실하거나 누락된 필드가 포함될 수 있다.
- 실제 Visit 등록 시에는 모든 필수 필드에 기존 Visit API의 검증 규칙을 적용한다.
- 파싱 결과를 Backend에서 재검증하며, AI가 반환한 Area와 점수를 신뢰하여 바로 저장하지 않는다.

## LINE Webhook API

LINE Bot의 메시지 이벤트를 수신하는 Backend Endpoint이다. 일반 사용자용 REST API가 아니며 LINE Platform만 호출한다.

```http
POST /api/line/webhook
X-Line-Signature: {signature}
Content-Type: application/json
```

### 수신 및 전달 규칙

- Backend는 UTF-8로 수신한 가공하지 않은 Request Body와 `LINE_CHANNEL_SECRET`을 사용해 `X-Line-Signature`를 HMAC-SHA256으로 검증한다. JSON 역직렬화는 서명 검증 후 수행한다.
- 서명이 없거나 유효하지 않으면 `401 Unauthorized`와 `INVALID_LINE_SIGNATURE`를 반환하며 이벤트를 처리하지 않는다.
- LINE Developers Console에서 Webhook과 Webhook Redelivery를 활성화한다.
- V1에서는 `LINE_ALLOWED_USER_ID`와 일치하는 1:1 대화의 텍스트 Message Event 및 확인·취소 Postback Event만 처리한다.
- 허용되지 않은 사용자, 그룹 대화, 텍스트가 아닌 메시지 및 지원하지 않는 이벤트는 외부에 정보를 노출하지 않고 `200 OK`로 종료한다.
- 이벤트가 없는 Webhook URL 검증 요청도 `200 OK`로 응답한다.
- 지원 이벤트는 최소 정보만 `line_webhook_event`에 먼저 저장하고 `webhookEventId` PK로 중복 수신을 방지한다. 원본 Webhook Body, Reply Token 및 Channel Access Token은 저장하지 않는다.
- Webhook 요청 안에서 OpenAI를 호출하지 않는다. DB Commit 후 이벤트마다 Cloud Tasks Task를 생성하고 안전한 전달이 확인된 경우에만 Response Body 없이 `200 OK`를 반환한다.
- Task 이름은 `webhookEventId`로부터 결정적으로 생성한다. 같은 Task가 이미 존재하면 이전 전달이 성공한 것으로 간주한다.
- DB 저장 또는 Task 전달에 실패하면 `503 Service Unavailable`과 `LINE_EVENT_DISPATCH_ERROR`를 반환해 LINE Webhook Redelivery 대상이 되게 한다.

### 비동기 처리 Endpoint

```http
POST /internal/tasks/line-events/{webhookEventId}
Authorization: Bearer {Cloud Tasks OIDC token}
```

- 일반 사용자 및 LINE Platform이 직접 호출하는 API가 아니다.
- Production에서는 Cloud Tasks 전용 Service Account가 발급한 OIDC Token의 서명, Issuer, Audience 및 Service Account를 검증한다.
- Cloud Tasks Header는 처리 정보로만 사용하며 호출자 인증 근거로 신뢰하지 않는다.
- Local에서는 `LocalLineEventDispatcher`, Production에서는 `CloudTasksLineEventDispatcher` 구현을 사용한다.
- 처리기는 `RECEIVED` 이벤트를 `PROCESSING`으로 원자적으로 전환하면서 `attemptCount`를 증가시키고 `processingStartedAt`을 기록한다.
- `COMPLETED` 이벤트가 재전달되면 작업을 반복하지 않고 `2xx`로 종료한다.
- `PROCESSING` 상태가 3분을 초과하면 이전 Process가 종료된 것으로 보고 다음 Cloud Tasks 시도가 이벤트를 다시 점유할 수 있다. 3분 이내의 중복 실행은 `5xx`로 종료해 나중에 재시도하게 한다.
- 일시적인 OpenAI, LINE API 또는 DB 오류가 발생하고 `attemptCount`가 5 미만이면 이벤트를 다시 `RECEIVED`로 전환하고 오류 코드를 기록한 후 `5xx`를 반환해 Cloud Tasks가 재시도하게 한다.
- 다섯 번째 애플리케이션 처리 시도에서도 실패하면, DB를 사용할 수 있는 경우 이벤트를 `FAILED`로 전환하고 마지막 오류 코드를 기록한다. 사용자 안내가 가능하면 실패 안내 Push Message를 Best-effort로 전송한 후 `2xx`로 종료한다.
- Cloud Tasks가 애플리케이션 Endpoint에 도달하지 못해 애플리케이션이 최종 상태를 기록할 수 없는 경우는 Cloud Tasks 최종 실패 로그와 `RECEIVED` 또는 Lease가 만료된 `PROCESSING` Row를 운영 점검 대상으로 관리한다.
- 재시도해도 해결되지 않는 입력 오류는 이벤트를 `FAILED`로 기록하고 사용자 안내가 가능하면 Push Message를 전송한 후 `2xx`로 종료한다.

### 텍스트 메시지 처리

- 텍스트는 `POST /api/visit-drafts`와 동일한 Application Service 및 검증 규칙으로 파싱한다.
- 처리 시작 시 `source_webhook_event_id`로 기존 Draft를 먼저 조회한다. 기존 Draft가 있으면 OpenAI를 다시 호출하거나 INSERT하지 않고 저장된 Draft와 warnings를 재사용한다.
- 파싱 결과는 `line_visit_draft`에 저장하며 생성 시점부터 24시간 동안 유효하다.
- Area, 방문일, 다섯 점수 및 memo를 보여주고 누락되거나 모호한 값은 임의로 보완하지 않고 `warnings`로 알린다.
- 필수 값이 모두 유효하면 Draft 상태를 `AWAITING_CONFIRMATION`으로 저장하고 확인 및 취소 Postback 버튼을 포함한 LINE Push Message를 전송한다.
- 필수 값이 누락되거나 유효하지 않으면 `NEEDS_INPUT`으로 저장하고, 확인 버튼 없이 부족한 값을 포함해 자연어 입력을 다시 보내도록 안내한다.
- 비동기 처리에서는 유효 시간이 짧은 Reply Token에 의존하지 않고 `line_user_id`를 대상으로 Push Message를 사용한다.

### Push Message 재시도

- 모든 Push Message는 최초 요청부터 `X-Line-Retry-Key`를 포함한다.
- Retry Key는 고정 Namespace 아래에서 `{webhookEventId}:{messagePurpose}`를 입력으로 사용하는 UUIDv5로 생성한다.
- `messagePurpose`는 `DRAFT_RESULT`, `CONFIRM_RESULT`, `CANCEL_RESULT`, `FAILURE_NOTICE` 중 하나를 사용해 한 이벤트의 서로 다른 메시지를 구분한다.
- 같은 Retry Key로 재시도할 때 수신자와 메시지 본문을 변경하지 않는다. Draft 결과는 저장된 Draft를 기준으로 동일하게 다시 구성한다.
- LINE API가 `409 Conflict`와 이미 수락된 Request ID를 반환하면 이전 Push가 성공한 것으로 간주한다.
- LINE의 Retry Key 유효 기간 안에서 재시도가 끝나도록 Cloud Tasks 재시도 기간을 24시간 미만으로 제한한다.

### 이벤트 완료 전환

- Draft 또는 Visit 처리 결과를 먼저 DB에 저장한 후 LINE Push Message를 전송한다.
- LINE API가 `2xx`를 반환하거나 이미 수락된 Request ID가 포함된 `409 Conflict`를 반환한 경우에만 `line_webhook_event`를 `COMPLETED`로 전환한다.
- Push 요청이 수락된 후 `COMPLETED` 저장에 실패하면 다음 Cloud Tasks 시도에서 저장된 Draft 또는 Visit 결과와 동일한 Retry Key를 사용해 같은 Push 요청을 재시도한다.
- Push 요청 전에 이벤트를 `COMPLETED`로 전환하지 않는다.

### 확인 및 취소 처리

- Postback Data는 `action=confirm&draftId={draftId}` 또는 `action=cancel&draftId={draftId}` 형식을 사용한다.
- Backend는 Postback을 보낸 LINE User와 Draft의 `line_user_id`가 같은지 확인한다. `draftId`만으로 소유 권한을 인정하지 않는다.
- 확인 시 Draft를 `SELECT ... FOR UPDATE`로 잠그고 상태, 24시간 만료, Area 존재 및 Soft Delete 여부, 날짜와 점수를 다시 검증한다.
- 유효한 Draft 확인은 Visit INSERT와 Draft의 `CONFIRMED` 전환 및 `confirmed_visit_id` 저장을 하나의 DB Transaction으로 처리한다.
- 이미 확인된 Draft의 확인 이벤트가 재처리되면 기존 `confirmed_visit_id`를 사용해 성공으로 처리하고 Visit을 중복 생성하지 않는다.
- `NEEDS_INPUT`, `CANCELLED` 또는 `EXPIRED` 상태는 확인할 수 없다.
- 취소는 `AWAITING_CONFIRMATION` Draft만 `CANCELLED`로 전환하며 반복 취소는 성공으로 간주한다.
- 완료 결과는 LINE Push Message로 전송한다.

### 보관 및 Logging

- 만료된 미확정 Draft는 조회 또는 확인 시 `EXPIRED`로 전환한다.
- `line_webhook_event`와 `line_visit_draft`는 생성 후 30일이 지나면 새 Webhook 처리 시 기회적으로 정리한다. 확인된 Visit은 삭제하지 않는다.
- LINE User ID, 메시지 원문, memo, Postback Data 및 Token은 로그에 기록하지 않는다.
- 로그에는 `webhookEventId`, `draftId`, 상태, 오류 코드, 재시도 횟수 및 처리 시간만 기록한다.


## Report API

### 리포트 유형

| 유형 | 역할 | Prompt Version |
|----|----|----|
| `SUMMARY` | SQL로 산출한 전체 통계와 항목별 Top 5를 중심으로 구성하고 AI는 짧은 Comment만 생성 | `summary-v1` |
| `ALL` | 모든 Area와 Visit 데이터를 기반으로 AI가 전체 지역 상세 분석 리포트 생성 | `all-v1` |
| `AREA` | 하나의 Area와 Visit 데이터를 기반으로 AI가 해당 지역을 상세 분석 | `area-v1` |
| `COMPARE` | 선택한 2개 이상 5개 이하의 Area를 AI가 항목별로 비교 분석 | `compare-v1` |

- 자연어 방문 평가 Parser의 Prompt Version은 `visit-parser-v1`을 사용한다.

### 리포트 생성
```http
POST /api/reports
```

```json

  "reportType": "SUMMARY"
}
```

#### ALL
```json
{
  "reportType": "ALL"
}
```

#### AREA
```json

}
```

```json

```

#### 검증 규칙
- SUMMARY, ALL: `areaIds` 생략. `null` 또는 빈 배열을 포함하여 필드가 전달되면 `400 Bad Request`
- AREA: `areaIds` 정확히 1개
- COMPARE: `areaIds` 2개 이상 5개 이하
- AREA 또는 COMPARE에서 `areaIds`가 누락되거나 `null` 또는 빈 배열이면 `400 Bad Request`
- 중복 ID는 허용하지 않으며 `400 Bad Request`
- 존재하지 않거나 삭제된 지역 ID가 있으면 `404 Not Found`
- 지원하지 않는 `reportType`이면 `400 Bad Request`

#### 대상 지역 저장 규칙
- Report 생성 시 분석 대상 Area를 `report_area` 연결 테이블에 저장한다.
- AREA는 1개, COMPARE는 요청 순서대로 2개 이상 5개 이하의 관계를 저장한다.
- ALL은 생성 당시 분석에 포함된 모든 Area 관계를 저장한다.
- SUMMARY는 SQL 전체 통계를 기반으로 하므로 Area 관계를 저장하지 않는다.
- `displayOrder`는 AREA와 COMPARE 요청의 지역 순서 및 ALL 생성 당시의 정렬 순서를 보존한다.

#### 생성 방식
- V1에서는 동기 방식으로 생성한다.
- OpenAI 호출, Markdown 생성, Cloud Storage 저장 및 Report 메타데이터 저장이 모두 완료된 경우에만 `201 Created`를 반환한다.
- OpenAI API 호출 실패 시 `502 Bad Gateway`를 반환한다.
- Cloud Storage 저장 실패 시 `503 Service Unavailable`을 반환한다.
- 생성 과정이 실패하면 Report 메타데이터를 저장하지 않는다.

#### Report ID 선점 및 실패 처리

객체 경로에 `reportId`가 포함되므로 다음 순서로 생성한다.

```text
요청 및 대상 데이터 검증
→ OpenAI 호출과 최종 Markdown 검증
→ DB Transaction 시작
→ storage_path가 NULL인 Report Row INSERT 및 AUTO_INCREMENT ID 획득
→ Report ID로 최종 객체 경로 생성
→ Report Storage에 Markdown 저장
→ Report.storage_path 갱신 및 ReportArea 저장
→ DB Transaction Commit
```

- `storage_path = NULL` 상태는 같은 Transaction 내부에서 ID를 얻기 위한 임시 상태이며 외부에 노출하거나 Commit하지 않는다.
- OpenAI 호출과 Markdown 검증은 불필요하게 DB Transaction을 오래 유지하지 않도록 Transaction 시작 전에 수행한다.
- Report Storage 저장 전에 실패하면 DB Transaction을 Rollback하며 저장 객체는 존재하지 않는다.
- Report Storage 저장에 실패하면 DB Transaction을 Rollback한다. 이때 사용되지 않은 AUTO_INCREMENT ID가 생길 수 있으며 정상적인 ID Gap으로 허용한다.
- Report Storage 저장 후 DB 갱신 또는 Commit에 실패하면 DB를 Rollback하고, 방금 저장한 객체를 Best-effort로 삭제한다.
- 보상 삭제도 실패하면 객체 경로와 오류를 로그에 남겨 운영 정리 대상으로 관리한다.
- DB Commit이 완료된 후에만 생성 성공으로 간주한다.

Location: /api/reports/10
```

  "model": "gpt-5.5-mini",
  "promptVersion": "compare-v1",

}
```

## Statistics API

### 집계 규칙

- 전체 평균은 삭제되지 않은 Area에 속한 모든 Visit을 같은 비중으로 계산한다.
- 지역별 평균은 해당 Area에 속한 모든 Visit을 같은 비중으로 계산한다.
- Top 5는 각 Area의 지역별 평균을 기준으로 산정한다.
- 평균은 소수점 둘째 자리에서 반올림하여 소수점 첫째 자리까지 반환한다.
- Top 5 대상 데이터가 없으면 빈 배열을 반환한다.
- Top 5는 점수 내림차순으로 정렬하고, 점수가 같으면 `areaId` 오름차순으로 정렬한다.
- Soft Delete된 Area와 해당 Area의 Visit은 전체 통계 및 Top 5에서 제외한다.

## Operations API

애플리케이션 운영 상태 확인을 위한 Spring Boot Actuator Endpoint이다. 사용자 기능 API가 아니며 Health Check 이외의 Actuator Endpoint는 외부에 노출하지 않는다.

### Liveness 조회

```http
GET /actuator/health/liveness
```

- Spring Boot 프로세스의 생존 상태만 확인한다.
- DB, OpenAI API 및 Cloud Storage 상태는 포함하지 않는다.
- 정상 상태는 `200 OK`, 비정상 상태는 `503 Service Unavailable`을 반환한다.

Response

```json
{
  "status": "UP"
}
```

### Readiness 조회

```http
GET /actuator/health/readiness
```

- Backend가 요청을 받을 준비가 되었는지 확인한다.
- Spring Boot Readiness 상태와 DB 연결 상태를 포함한다.
- OpenAI API와 Cloud Storage 상태는 포함하지 않는다.
- 정상 상태는 `200 OK`, 준비되지 않은 상태는 `503 Service Unavailable`을 반환한다.

Response

```json
{
  "status": "UP"
}
```

- 두 Endpoint는 Cloud Run Probe 호출을 위해 인증 없이 접근할 수 있다.
- 응답에는 Component 상세 정보, DB 주소 및 내부 오류 내용을 노출하지 않는다.
