## Design Notes

- Visit는 Area를 참조한다. (1:N)
- AI Report는 Cloud Storage에 저장하고 DB에는 `reports/v1/{reportType}/{filename}_{yyyy-MM-dd}_{reportId}.md` 형식의 객체 경로만 저장한다.
- Visit 데이터는 Source of Truth로 간주한다.
- AI는 데이터를 생성하지 않고, 데이터를 해석한다.
- Area는 `deleted_at`을 사용해 Soft Delete한다.
- Area의 `(prefecture, city, name)` 조합은 UNIQUE이며 `station`은 중복 판단에서 제외한다.
- ReportArea는 Report와 생성 대상 Area를 연결한다. 두 FK를 복합 PK로 사용하고 `display_order`로 리포트 내 지역 순서를 보존한다.
- ReportArea는 Visit 등록 시점이 아니라 Report 생성 시점에만 저장한다.
- AREA는 1건, COMPARE는 2~5건, ALL은 생성 당시 모든 Area 관계를 저장하며 SUMMARY는 관계를 저장하지 않는다.
- Entity PK는 `BIGINT AUTO_INCREMENT`를 사용한다. `report_area`는 두 FK를 복합 PK로 사용하므로 별도 ID를 생성하지 않는다.
- Visit의 다섯 점수는 DB와 Backend 모두에서 0 이상 10 이하로 검증한다.
- `visit_date`는 사용자가 방문한 날짜이므로 시각 없이 `DATE`로 저장한다.
- `created_at`, `updated_at`, 만료 및 처리 시각은 정렬·감사·재시도 판단에 필요하므로 초 단위 `TIMESTAMP`로 저장한다. 밀리초는 사용하지 않는다.
- DB와 애플리케이션은 UTC를 사용한다. API는 UTC ISO8601 형식으로 반환하고 Frontend가 표시할 때 사용자 시간대로 변환한다.
- `report.prompt_version`은 `visit-parser-v1`처럼 10자를 넘는 버전을 보존할 수 있도록 `VARCHAR(30)`을 사용한다.
- `report.storage_path`는 Report ID 선점 과정에서 같은 DB Transaction 안에서만 일시적으로 `NULL`일 수 있다. Transaction이 Commit된 Report에는 반드시 최종 경로가 존재하도록 Backend가 보장한다.
- `line_webhook_event`는 LINE 이벤트의 최소 처리 정보와 상태를 저장하고 `webhook_event_id`를 PK로 사용해 중복 수신을 방지한다.
- `line_webhook_event.processing_started_at`은 작업 중 Process 종료를 감지하는 처리 Lease 기준으로 사용한다.
- `line_visit_draft`는 LINE 자연어 입력의 파싱 결과를 최대 24시간 보관한다. 필수 값이 모두 유효한 Draft만 확인할 수 있다.
- Draft 확인 시 Visit 저장과 Draft의 `CONFIRMED` 전환을 하나의 DB Transaction으로 처리하며 `confirmed_visit_id`로 결과를 보존한다.
- `line_visit_draft.source_webhook_event_id`는 UNIQUE이며 재처리 시 기존 Draft를 조회해 재사용한다.
- Draft가 `CONFIRMED`이면 `confirmed_visit_id`가 반드시 존재하고, 그 외 상태에서는 `NULL`이어야 한다.
- LINE User ID는 Draft 소유자 검증과 Push Message 전송에만 사용하고 API 응답 및 로그에 노출하지 않는다.
- LINE 이벤트 및 Draft는 생성 후 30일이 지나면 새 Webhook 처리 시 기회적 정리 대상으로 삼는다.
- `town-ai-v1.sql`을 현재 스키마 정의 기준으로 사용한다. PNG와 XLSX는 다음 다이어그램 갱신 시 동기화한다.
